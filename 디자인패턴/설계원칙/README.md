# 디자인 패턴 - 설계 원칙

## 디자인 패턴의 분류

| 구분                       | 설명                    | 예시                      |
| -------------------------- | ----------------------- | ------------------------- |
| **생성 패턴 (Creational)** | 객체 생성 방식을 추상화 | Singleton, Factory Method |
| **구조 패턴 (Structural)** | 객체 간 관계 구성       | Adapter, Bridge           |
| **행위 패턴 (Behavioral)** | 객체 간 상호작용 정의   | Template Method, Strategy |

## SOLID 원칙

- 객체지향 설계 원칙(5가지)
- 클래스와 모듈이 변경에 강하고, 확장에 유연하도록 만든다.
- **Single Responsibility(단일 책임 원칙)**
  - 하나의 클래스는 단 하나의 책임만 가져야 한다.
  - 클래스를 변경하는 이유는 단 하나여야 한다.
  - 책임이 분리되면 코드의 가독성과 유지보수성이 향상됨
- **Open-Closed(개방-폐쇄 원칙)**
  - 확장에는 열려있고, 변경에는 닫혀있어야 한다.
  - 기존 코드를 변경하지 않으면서 기능을 추가할 수 있어야 한다.
  - 추상화와 다형성을 활용
- **Liskov Substitution(리스코프 치환 원칙)**
  - 자식 클래스는 부모 클래스를 대체할 수 있어야 한다.
  - 상속 관계에서 부모 클래스의 인스턴스를 자식 클래스의 인스턴스로 치환해도 프로그램의 동작이 정상적이어야한다.
  - IS-A(상속)관계가 올바르게 성립해야 한다.
- **Interface Segregation(인터페이스 분리 원칙)**
  - 클라이언트는 자신이 사용하지 않는 메소드에 의존하지 않아야 한다.
  - 범용 인터페이스보다 구체적인 여러 인터페이스가 낫다.
  - 인터페이스를 작고 구체적으로 분리
- **Dependency Inversion(의존성 역전 원칙)**
  - 구체적인 것이 아닌 추상적인 것에 의존해야한다.
  - 구체 클래스보다 추상클래스 또는 인터페이스에 의존

## GRASP

- 객체 간 책임 배분 원칙(9가지)
- **Information Expert (정보 전문가)**
  - **책임을 수행하는데 필요한 정보를 가진 객체에게 책임을 할당**
  - 정보를 가진 객체가 그 정보를 사용하는 작업을 수행
  - 응집도를 높이고 결합도를 낮춤
- **Creator (생성자)**
  - **객체 생성 책임을 적절한 클래스에 할당**
  - 다음 조건 중 하나를 만족하는 클래스가 생성 책임을 가짐
    - 생성될 객체를 포함하거나 집합
    - 생성될 객체를 기록
    - 생성될 객체를 긴밀하게 사용
    - 생성될 객체의 초기화 데이터를 가지고 있음
- **Controller (컨트롤러)**
  - **시스템 이벤트를 처리할 책임을 비UI 객체에게 할당**
  - 시스템 작업을 처리하는 첫 번째 객체
  - UI와 도메인 로직을 분리함
- **Low Coupling (낮은 결합도)**
  - **클래스 간의 의존성 최소화**
  - 변경의 영향을 줄이고 재사용성을 높임
  - 인터페이스를 통한 의존성 주입 활용
- **High Cohesion (높은 응집도)**
  - **클래스의 책임이 밀접하게 관련되어 있고 집중되어야 함**
  - 한 클래스가 하나의 명확한 목적을 가짐
  - 이해하기 쉽고 유지보수가 용이함
- **Polymorphism (다형성)**
  - **타입에 따라 행동이 달라지는 경우 다형성을 사용함**
  - 조건문 대신 다형성 활용
  - 확장성과 유연성 향상
- **Pure Fabrication (순수 가공)**
  - **도메인 개념이 아니지만 높은 응집도와 낮은 결합도를 유지하기 위해 인위적으로 만든 클래스**
  - 특정 책임을 위해 설계된 헬퍼 클래스
  - Information Expert 원칙을 위배하더라도 설계 품질 향상을 위해 사용
- **Indirection (간접 참조)**
  - **두 객체 사이의 직접적인 결합을 피하기 위해 중간 객체를 도입함**
  - 중재자를 통해 낮은 결합도 유지
  - 변경의 영향을 최소화
- **Protected Variations (변경 보호)**
  - **변경될 가능성이 있는 지점을 식별하고 안정된 인터페이스로 감싼다**
  - 변경의 영향이 다른 부분으로 전파되지 않도록 보호
  - OCP(개방-폐쇄 원칙)와 유사한 개념
